## Overview

The MCP client module handles all communication with the [thunderbird-mcp](https://github.com/joelpurra/thunderbird-mcp) extension running inside Thunderbird. It lives in `archive_marketing.py` from line ~564 to ~743.

The client is intentionally minimal: a single `_mcp_call()` function wraps all HTTP communication, and three domain-specific functions (`fetch_page`, `move_emails`, and the report functions) use it to issue named MCP tool calls.

**All network communication is localhost-only.** No external network access occurs.

---

## Protocol

The thunderbird-mcp extension exposes a **JSON-RPC 2.0 over HTTP** interface:

```
POST http://127.0.0.1:{port}/
Content-Type: application/json
Authorization: Bearer {token}

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "toolName",
    "arguments": { ... }
  }
}
```

Successful response envelope:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{ ... JSON-encoded result ... }"
      }
    ]
  }
}
```

Error response:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32600,
    "message": "Invalid parameters"
  }
}
```

The result payload is **double-encoded**: the outer envelope contains a `content` array with a `text` field whose value is a JSON string. `_mcp_call()` handles this double-parsing transparently.

## Authentication

Authentication uses a **Bearer token** passed as an HTTP header:

```
Authorization: Bearer eyJhbGciOiJIUzI1...
```

The token is generated by Thunderbird when the thunderbird-mcp extension starts. It is valid for the lifetime of the current Thunderbird session. If Thunderbird restarts, a new token is generated and written to `connection.json`.

The token is read at runtime from the connection file and held in memory for the duration of the script run. It is never logged, written to disk, or transmitted outside localhost.

## Connection File

### Location

| Platform | Default path |
|----------|-------------|
| Windows | `%TEMP%\thunderbird-mcp\connection.json` |
| macOS/Linux | `/tmp/thunderbird-mcp/connection.json` |

Override with `--connection-file` or `ARCHIVE_CONNECTION_FILE`.

### Format

```json
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "port": 8765
}
```

### Security

The connection file contains a live session token. The script calls `validate_connection_file()` after reading it to warn if the file has group-readable or world-readable permissions on POSIX systems:

```
[WARN] Connection file has permissive permissions: 0o644
       Run: chmod 600 /tmp/thunderbird-mcp/connection.json
```

The file is excluded from git via `.gitignore`.

## Functions

### `_mcp_call()`

```python
def _mcp_call(token: str, port: int, tool_name: str, arguments: Dict) -> object
```

Core HTTP request function. Sends a single JSON-RPC `tools/call` request and returns the parsed result.

**Request construction:**
- URL: `http://127.0.0.1:{port}/`
- Method: `POST`
- Timeout: 30 seconds
- Content-Type: `application/json`
- Body: JSON-encoded JSON-RPC 2.0 envelope
- A global `_req_id` counter increments with each call for request correlation

**Response handling:**
1. HTTP response is JSON-parsed
2. If `response["error"]` is set → raises `RuntimeError("MCP error {code}: {message}")`
3. Extracts `result.content[0].text` and JSON-parses it again
4. If the response shape is unexpected → raises `RuntimeError`

**Raises:**
- `RuntimeError` — MCP-level errors or unexpected response shapes
- `urllib.error.URLError` — network errors (caller is responsible for retries)

---

### `load_connection()`

```python
def load_connection(connection_file: str) -> Tuple[str, int]
```

Reads and validates the connection file. Called once at the start of `run()`.

On failure, calls `sys.exit()` with a user-friendly checklist rather than raising an exception — this is appropriate since a missing connection file is always a fatal configuration error.

### `fetch_page()`

```python
def fetch_page(
    token: str, port: int, inbox: str,
    offset: int, page_size: int, days_back: int,
) -> Tuple[List[Dict], int]
```

Wraps the `getRecentMessages` MCP tool call. Returns `(messages, total_count)`.

The MCP tool returns messages sorted newest-first. The `offset` parameter enables pagination — each page call increments offset by `page_size`.

**Response handling variants:**
- If result is `dict` with `"messages"` key: returns `result["messages"], result["totalMatches"]`
- If result is a plain `list`: returns `result, 0` (older thunderbird-mcp versions)
- Otherwise: returns `[], 0`

---

### `move_emails()`

```python
def move_emails(
    token: str, port: int, inbox: str,
    message_ids: List[str], destination: str,
) -> int
```

Wraps the `updateMessage` MCP tool call to perform bulk IMAP moves.

The `move_batch` setting (default: 50) limits how many IDs are sent per call. `run()` handles the batching loop; `move_emails()` handles a single batch.

## Error Handling

### Fetch retries

`run()` wraps each `fetch_page()` call with a retry loop (3 attempts):

```python
for attempt in range(3):
    try:
        emails, total = fetch_page(...)
        break
    except Exception as exc:
        if attempt == 2:
            print(f"[ERROR] Failed after 3 attempts: {exc}")
            break
        wait = 10 * (attempt + 1)  # 10s, 20s
        time.sleep(wait)
```

### Move errors

`move_emails()` errors are caught at the call site in `run()` and logged with `[ERROR]`. Execution continues with the next batch — partial failures do not abort the run.

### Report errors

`send_report_email()` and `cleanup_prev_reports()` catch all exceptions internally and print warnings. A report failure never causes the main run to fail.

### Connection errors

`load_connection()` calls `sys.exit()` on missing/malformed files. There is no retry for the connection file — it is a startup check.

## Rate Limiting

The thunderbird-mcp bridge delegates to Thunderbird's IMAP client, which is subject to server-side rate limits. Two configurable delays prevent overwhelming the IMAP server:

| Setting | Default | Applied between |
|---------|---------|----------------|
| `fetch_delay` | `2.0s` | Page fetches (every 100 emails) |
| `move_delay` | `1.5s` | Batch move calls |

For Gmail accounts with a large inbox, increasing `fetch_delay` to `3.0` or higher is recommended to avoid temporary IMAP throttling (`[IMAP] Too many simultaneous connections`).
